# PracticeHell 最小実装計画

## 目的

解答者が参加URLから入り、出席情報を登録し、LLM生成問題へ連続して解答できる最小のアプリを作る。

## 対象範囲

### 最初に実装するもの

- `/?code=<参加コード>` から問題YAMLを特定する。
- 出席番号と名前を入力して演習を開始する。
- 文章と式からなる数値解答問題を1問ずつ表示する。
- 現在の問題を表示している間に、次の1問をLLM APIで先行生成する。
- 解答直後に正誤、正答、総解答数、EMA、達成状況を表示する。
- 最低解答数とEMA閾値を満たした後も、演習を続けられるようにする。
- 出席情報、生成問題、提出値、正誤、解答時点の進捗をSQLiteへ保存する。
- 白背景の最小限の画面と、差し替えやすい単一CSSファイルを用意する。

### 後回しにするもの

- 教師用のリアルタイム進捗画面
- 問題YAMLを作成する管理画面や生成ツール
- 教師・解答者のアカウント認証
- 画像、音声、選択式、自由記述式の問題
- 複数台構成や高度な運用機能

## データと動作

- 問題YAMLは起動時に読み込み、`join_code` の重複や必須項目の不足があれば起動エラーにする。
- 参加コードはURLで安全に扱える英数字・ハイフン・アンダースコアに限定する。
- 出席番号と名前を登録するたびに新しい演習セッションを作る。再開機能は初期版に含めない。
- LLMは `question_text` と `correct_answer` からなる構造化データを返す。モデル名とAPIキーは環境変数で設定する。
- 生成済み問題はSQLiteに `ready` として保存し、表示時に `presented`、解答後に `answered` とする。
- 次問が未生成の場合だけ待機表示を出し、生成失敗時は同じ位置から再試行できるようにする。
- 数値採点は、問題YAMLの絶対許容誤差と相対許容誤差の大きい方を使用する。
- EMAの初回値は最初の正誤とし、以降はYAMLの `ema_alpha` で更新する。

## 問題YAML

[problems/example.yaml](problems/example.yaml) を初期テンプレートとする。

- アプリへの参加情報: `join_code`、`title`
- LLMへの生成指示: `generation.prompt`
- 解答と採点: `question.answer_type`、許容誤差
- 習熟基準: `mastery.minimum_answers`、`threshold`、`ema_alpha`

問題YAMLの生成方法は、この形式を実際に数種類使ってから決定する。

## 実装順

1. YAML読込・検証と参加コード照合
2. 出席情報入力とSQLite保存
3. 固定問題による出題・採点・EMA更新
4. LLM APIによる構造化問題生成
5. 次の1問の先行生成と失敗時の再試行
6. 一連のブラウザ動作テスト
